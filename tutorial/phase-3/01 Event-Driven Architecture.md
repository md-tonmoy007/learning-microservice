# 01 Event-Driven Architecture

> Phase 3 replaces the direct HTTP call from api-gateway to orchestrator with Kafka events, decoupling the services and enabling real-time progress streaming.

Related: [[Kafka and Redpanda]] В· [[Redis]] В· [[Server-Sent Events]] В· [[Home]]

---

## The Problem with Phase 2

In Phase 2, submitting a research query works like this:

```
User в†’ POST /research в†’ api-gateway в†’ HTTP POST в†’ orchestrator в†’ starts LangGraph
                                                     в†“ waits for it to finish
                                         returns task_id when done starting
```

The api-gateway **blocks** waiting for the orchestrator to respond. If the orchestrator is slow to start, the user waits. If the orchestrator is down, the gateway returns 503. They are **tightly coupled**.

More importantly, the user has no way to see live progress. They can only poll `GET /research/{id}/status` and wait for `"completed"`. They don't know if the planner ran, how many search results came back, or when summarization starts.

---

## The Phase 3 Solution

```
Phase 2 (synchronous):
  api-gateway в”Ђв”ЂHTTPв”Ђв”Ђв–є orchestrator в”Ђв”Ђв–є (LangGraph runs) в”Ђв”Ђв–є response

Phase 3 (event-driven):
  api-gateway в”Ђв”ЂKafkaв”Ђв”Ђв–є [research.created] в”Ђв”Ђв–є orchestrator
  api-gateway returns immediately with task_id

  orchestrator в”Ђв”ЂKafkaв”Ђв”Ђв–є [research.planned] в”Ђв”Ђв–є api-gateway SSE в”Ђв”Ђв–є User browser
  orchestrator в”Ђв”ЂKafkaв”Ђв”Ђв–є [research.search.completed] в”Ђв”Ђв–є SSE в”Ђв”Ђв–є User browser
  ...
  orchestrator в”Ђв”ЂRedisв”Ђв”Ђв–є task:{id}:status = "completed"
```

Three channels now carry information:

| Channel | Direction | Used for |
|---------|-----------|---------|
| Kafka `research.created` | gateway в†’ orchestrator | kick off a workflow |
| Kafka `research.*` progress | orchestrator в†’ gateway | live progress events |
| Redis `task:{id}:status` | orchestrator в†’ gateway | fast status checks |

---

## Why This Is Better

**Decoupling** вЂ” the gateway doesn't know or care whether the orchestrator is running. It publishes to Kafka and returns. If the orchestrator restarts mid-flight, Kafka holds the message; the orchestrator picks it up when it comes back.

**Real-time feedback** вЂ” users see "planning done", "searching", "summarizing" as they happen instead of waiting for a final answer.

**Scalability** вЂ” you can run multiple orchestrator instances. Kafka distributes `research.created` messages across them (consumer group load balancing).

**Resilience** вЂ” if the orchestrator crashes after consuming a `research.created` message but before finishing, the message is not acknowledged. Kafka can be configured to redeliver it (at-least-once delivery). In Phase 3 we use at-most-once for simplicity, but the groundwork is there.

## Key Design Decisions

### task_id is generated by the gateway, not the orchestrator

In Phase 2, the orchestrator created the task and returned its DB-generated `id`. In Phase 3, the gateway generates the `task_id` (`uuid4()`) **before** publishing the Kafka event. This means:
- The user gets their `task_id` immediately
- The SSE stream URL (`/research/{task_id}/events`) is known before the orchestrator even receives the message
- The orchestrator inserts the task using the provided `task_id` instead of auto-generating one

### LangGraph switches from `ainvoke` to `astream`

In Phase 2, `research_graph.ainvoke(state)` runs everything and returns the final state at the end. In Phase 3, `research_graph.astream(state, stream_mode="updates")` yields a partial dict after each node completes. This lets `run_workflow` publish a Kafka event between nodes without waiting for the whole workflow to finish.

### Redis is write-ahead for status

The orchestrator writes to Redis at the start of each step, before the step runs. So `GET /research/{id}/status` always reflects the most recently **started** step, not the most recently **completed** one. This is intentional вЂ” users want to know "what is happening now", not "what finished last".
