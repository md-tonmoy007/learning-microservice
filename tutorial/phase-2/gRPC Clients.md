# gRPC Clients

> The orchestrator's `grpc_clients/` folder contains thin async wrappers that open a gRPC channel, call one RPC method, and return plain Python dicts — keeping the LangGraph nodes free of gRPC details.

Related: [[gRPC]] · [[Protocol Buffers]] · [[gRPC aio Servers]] · [[LangGraph Phase 2]] · [[Planner Agent]] · [[Search Agent]] · [[Summarizer Agent]] · [[Critic Agent]] · [[Report Service]] · [[Home]]

## The Code

Five files, one per agent. They all follow the same three-step pattern:

```
grpc_clients/
  planner.py      → create_plan(task_id, user_query) → dict
  search.py       → search_queries(task_id, queries) → list[dict]
  summarizer.py   → summarize_search_results(...) → dict
  critic.py       → critique_summary(...) → dict
  report.py       → generate_final_report(...) → str
```

### `planner.py` — the full pattern

```python
import grpc
from app.core.config import settings
from app.grpc_generated_path import ensure_grpc_generated_on_path

ensure_grpc_generated_on_path()
from app.grpc_generated import planner_pb2, planner_pb2_grpc


async def create_plan(task_id: str, user_query: str) -> dict:
    async with grpc.aio.insecure_channel(settings.planner_agent_address) as channel:
        stub = planner_pb2_grpc.PlannerServiceStub(channel)
        try:
            response = await stub.CreatePlan(
                planner_pb2.PlanRequest(task_id=task_id, user_query=user_query)
            )
        except grpc.aio.AioRpcError as exc:
            raise RuntimeError(f"Planner gRPC failed: {exc.details()}") from exc

    return {
        "search_queries": list(response.search_queries) or [user_query],
        "report_sections": list(response.report_sections),
    }
```

### `summarizer.py` — building proto messages from dicts

```python
async def summarize_search_results(task_id, user_query, results: list[dict]) -> dict:
    request_results = [
        summarizer_pb2.SearchResult(
            url=str(result.get("url", "")),
            content=str(result.get("content", "")),
        )
        for result in results
    ]

    async with grpc.aio.insecure_channel(settings.summarizer_agent_address) as channel:
        stub = summarizer_pb2_grpc.SummarizerServiceStub(channel)
        try:
            response = await stub.Summarize(
                summarizer_pb2.SummarizeRequest(
                    task_id=task_id, user_query=user_query, results=request_results,
                )
            )
        except grpc.aio.AioRpcError as exc:
            raise RuntimeError(f"Summarizer gRPC failed: {exc.details()}") from exc

    return {
        "summary": response.summary,
        "key_points": list(response.key_points),
        "citations": list(response.citations),
    }
```

## Walkthrough

### `async with grpc.aio.insecure_channel(...) as channel`

This is the gRPC channel — a persistent connection to the target service. `insecure_channel` means no TLS (fine inside a Docker network). Using it as a context manager (`async with`) ensures the connection is properly closed after the RPC call completes.

Every function in `grpc_clients/` creates a **new channel per call**. This is simple but not optimal: in production you'd create one shared channel at startup and reuse it across calls (channel pooling). For Phase 2's learning goal, one-channel-per-call is correct and readable.

### `planner_pb2_grpc.PlannerServiceStub(channel)`

The stub is the **client-side proxy**. It has one method per RPC defined in the proto: `stub.CreatePlan(...)`, `stub.Search(...)`, etc. Calling `stub.CreatePlan(request)` serializes the request to binary, sends it over the channel, waits for the response, deserializes it, and returns the response message object.

You don't write any of this — it's all generated by `grpc_tools.protoc` into `planner_pb2_grpc.py`.

### `grpc.aio.AioRpcError`

When something goes wrong on the server side (the agent calls `context.abort(INTERNAL, "msg")`), the stub raises `grpc.aio.AioRpcError`. This is **not** a regular Python exception — it's gRPC's error type. It has:

```python
exc.code()      # grpc.StatusCode.INTERNAL
exc.details()   # "LLM unavailable" — the string from context.abort()
```

Every client wraps the stub call in `try/except grpc.aio.AioRpcError` and converts it to a plain `RuntimeError`. This keeps LangGraph nodes from needing to know about gRPC error types.

### Converting Proto Types to Python Dicts

Proto `repeated` fields are not regular Python lists — they're `RepeatedScalarContainer` objects. Always convert with `list(...)` before returning:

```python
return {
    "search_queries": list(response.search_queries),  # ← important
}
```

This matters because LangGraph state is a `TypedDict` with `list[str]` type hints. If you pass a `RepeatedScalarContainer` instead of a `list`, type checking tools will flag it and serialization might behave unexpectedly.

### Building the Summarizer's Input

The summarizer client is the most interesting because it has to build **proto message objects** from plain dicts:

```python
request_results = [
    summarizer_pb2.SearchResult(
        url=str(result.get("url", "")),
        content=str(result.get("content", "")),
    )
    for result in results
]
```

The `state["search_results"]` list contains plain Python dicts `{"title": ..., "url": ..., "content": ...}`. These have to be converted to `summarizer_pb2.SearchResult` proto message objects before they can be included in a `SummarizeRequest`. The client only passes `url` and `content` — `title` and `source_type` aren't needed for summarization.

## Service Address from Settings

Each client reads the target address from `settings`:
```python
settings.planner_agent_address    # "planner-agent:50051"
settings.search_agent_address     # "search-agent:50052"
settings.summarizer_agent_address # "summarizer-agent:50053"
settings.critic_agent_address     # "critic-agent:50054"
settings.report_service_address   # "report-service:50055"
```

These default to the Docker Compose service names. Override them with env vars (`PLANNER_AGENT_ADDRESS=...`) for local development where services run on `localhost`.

## The `grpc_generated_path.py` Utility

Each service has a `grpc_generated_path.py` file that adds `app/grpc_generated/` to `sys.path`:

```python
def ensure_grpc_generated_on_path() -> None:
    generated_dir = str(Path(__file__).parent / "grpc_generated")
    if generated_dir not in sys.path:
        sys.path.insert(0, generated_dir)
```

This is needed because generated stubs are in `app/grpc_generated/` but that directory has no `__init__.py` (it's gitignored, so we can't commit package markers). Adding it to `sys.path` makes `from app.grpc_generated import planner_pb2` work regardless.

> [!tip]
> The boundary between the gRPC client and the LangGraph node is clean: nodes call Python functions with Python types, get Python dicts back. gRPC, proto messages, and stubs are all hidden inside `grpc_clients/`. If we ever replace gRPC with REST in a future refactor, only `grpc_clients/` would need to change — not the nodes.
